run_find_boundary()
Reads hex stats, filters serviceable hexes, DBSCAN-clusters them per partner, dissolves hex polygons into boundary shapes. Output is partner_cluster_boundaries.h5 — the df_bound input to process().
Why boundaries exist: Partners serve thousands of hex cells, but we need a single "coverage shape" per partner to answer geometric questions — is this lead inside the service area? How far from the edge? How many partners overlap here? Boundaries are the macro envelope; hexes are the micro grid inside it.

1Load poly_stats.h5 — every hex with SE, color, installs. Filter to lightgreen + orange only (crimson = unserviceable).
2Installs floor — drop hexes below p30 installs. Keeps only hexes with enough evidence to matter.
3DBSCAN per partner — haversine metric on hex centroids. eps = best_size × 2 (adaptive per partner). min_samples=2. Groups nearby hexes into clusters.
4Noise rescue — DBSCAN labels isolated hexes as noise (−1). High-value singletons with installs ≥ p90 are rescued as their own p90_single_cluster.
5Dissolve — unary_union merges all hex polygons in a cluster into one boundary shape. Buffer trick (+1e-9 then −1e-9) fills tiny gaps between hexes.
6Compute metadata — centroid, projected area (UTM 43N), total_installs, n_hexes, cluster_type. Save as partner_cluster_boundaries.h5.
Buffer trick explained: Adjacent hexes touch at edges but may have infinitesimal gaps due to floating-point coordinates. buffer(1e-9) expands each hex by a nanometer, closing gaps. unary_union then merges them into one polygon. buffer(-1e-9) shrinks back to the original size. Without this, the union would produce a MultiPolygon with hairline cracks.
Why rescue p90 singletons? — A hex with 200 installs that happens to be geographically isolated (no neighbour within eps) would be discarded by DBSCAN as noise. But it's a proven high-performance area — losing it means losing a partner's best territory from the boundary map. The p90 rule rescues these as p90_single_cluster (tagged separately so downstream can use is_solo_cluster).

add_boundary_details_precise()
Core question: "How deep inside a partner's service area is this lead?" — A lead near the edge of a boundary is riskier (the partner may not serve it well). A lead deep inside is safer. This function measures that geometric relationship.

    df["dist_to_boundary_edge_m"] < 0.7 * df["dist_to_cluster_center_m"],

5depth_score = dist_center − dist_edge. Positive = deep inside, negative = near edge.

A hex grid can't tell you "this lead is 50m from the edge of the entire coverage zone" because hexes tile uniformly and don't know about the overall boundary contour. depth_score, near_edge, nmbr_boundaries_within_100m — these are all geometric position features relative to the partner's outer envelope, not local conversion features.

compute_contested_metrics_engine()
Core question: "Where two or more partners overlap, how does that shared zone perform?" — If the intersection zone has high SE, the lead is in well-served territory regardless of which partner gets it. If SE is low, it's contested and underserved. The contested_field adds spatial smoothness via inline KDE.

HEX CONSENSUS TRACK
get_parent_hexagon()
Core question: "What's the best estimate of installation probability at this location, given all partners whose hexes cover it?" — A single target may sit inside hexes from 3 different partners, each with different SE. We can't just pick one — we need a weighted consensus that respects evidence (total decisions) and applies shrinkage so low-evidence hexes don't dominate.
🔬
Why do we need hex consensus when boundary + contested already exist?
They answer fundamentally different questions at different spatial scales:
📐
Concrete example of why all three matter:
A lead sits 200m from the boundary edge (boundary says: risky, near_edge=1), in an overlap zone with contested_se=0.35 (contested says: the shared zone is mediocre). But the specific hex covering it from Partner C has SE=0.55 with 300 decisions, and its hop-1 neighbours average 0.48 (hex says: actually this micro-area is strong, and the neighbourhood confirms it). Without hex consensus, you'd miss this — boundary features only see the macro shape, contested only sees the overlap zone's aggregate, neither gives you the per-partner, fine-grained, statistically-shrunk conversion rate at the exact grid cell level. That's what drives allocation: which partner has the best local track record here. 
# A.3: Asymmetric credibility shrinkage
# Low-evidence hexes get pulled toward zero (conservative)
# Negative SE gets amplified (penalize bad areas more)
ratio = np.maximum(
    config.MIN_SHRINKAGE_RATIO,
    joined["total"] / (joined["total"] + config.SHRINKAGE_K)
)
joined["se_shrunk"] = np.where(
    joined["se"] >= 0,
    joined["se"] * ratio,    # positive: dampen
    joined["se"] / ratio,    # negative: amplify
)

# A.4: Evidence-weighted intermediate for consensus
joined["install_shrunk"] = joined["se_shrunk"] * joined["total"]

# A.5: Mobile-level groupby → divide back by total
consensus["weighted_se_shrunk"] = (
    consensus["install_shrunk"] / consensus["total"]
)
Shrink then aggregate — Shrinkage happens per-hex before the consensus groupby. This ensures a hex with 5 decisions can't dominate over one with 500. The ratio is total / (total + K): if K=100 and a hex has 10 decisions, its SE gets multiplied by 0.09. A hex with 1000 decisions keeps ~0.91 of its SE. Asymmetry means bad areas get amplified — cautious by design.

compute_hop_features()
Core question: "What do the neighbours of this hex look like?" — A hex might have SE=0.5, but if all its neighbours have SE=0.1, it's an island of success about to regress. Hop features capture this spatial context via 3 concentric rings of neighbours, weighted by inverse distance. The gradient (hop1 → hop3) tells you if performance decays or improves as you move outward.

ALL-HEX COMBINED FIELD
Per-hex Gaussian Field → Mobile Aggregation
Core question: "What's the local sentiment around this lead — are nearby historical decisions positive or negative?" — Unlike SE (which is a hex-level ratio), the Gaussian field is a continuous, spatially-smooth surface. It accounts for the exact distance to each historical decision point and how recent that decision was. By computing it per-hex and then aggregating, we get a field value that reflects all partners who cover this location — not just the best or worst one.

Why kernel_sum-weighted? — kernel_sum = Σ(k) is how many nearby source points contributed to this hex's field estimate (weighted by spatial proximity). A hex where the target is surrounded by dense source data gets higher kernel_sum → more influence in the final consensus. This is analogous to precision-weighting: higher confidence estimates dominate.

compute_adaptive_gaussian_field()
source_coords_rad = np.radians(df_source[["latitude", "longitude"]])
tree = BallTree(source_coords_rad, metric="haversine")

for i, (target_coord, eval_time) in enumerate(...):
    # Radius query: O(1) memory per target
    indices, dist_rad = tree.query_radius(
        target_coord, r=radius_rad, return_distance=True
    )
    dist_m = dist_rad[0] * EARTH_RADIUS_M

    # Causal: only use past observations
    ages_days = (eval_time - neigh_times) / np.timedelta64(1, "D")
    time_weights[past_mask] = np.exp(-lambda_decay * ages_days[past_mask])

    # Adaptive bandwidth: each source has its own h
    kernel = np.exp(-0.5 * (dist_m**2) / (neigh_h**2))

    # Weighted NW estimator
    predicted[i] = np.sum(kernel * effective_weights) / (np.sum(kernel) + 1e-12)
    kernel_sums[i] = np.sum(kernel)  # confidence measure
Adaptive bandwidth h — Each source point has its own bandwidth, unlike standard KDE with a single global h. In dense urban areas, h is small (tight influence radius). In sparse areas, h is large (spread influence further). This prevents over-smoothing in cities while still getting signal in thin areas. The causal mask ensures we never leak future information — the field at time T only sees decisions before T.

GEOMETRIC PATTERN FEATURES
batch_compute_geometry() + regime derivation
Runs outside process(), in step2_score_test.py. For each test lead, finds historical decision points within GEOM_SEARCH_RADIUS_M (100m) via BallTree, then computes shape descriptors (anisotropy, density, convex hull, spread) of that point cloud. Three logistic regime scores classify the local pattern — dense blob, linear gully, or sparse frontier. These features enrich df_test before it enters process().
Core question: "What does the shape of nearby history look like?" — Two leads with identical SE can differ wildly: one surrounded by a tight, round cluster of installs (dense blob — high confidence), the other along a narrow road with points only in a line (gully — uncertain perpendicular reach). This feature captures the geometric morphology of the local point cloud, which is invisible to SE, KDE fields, and boundary distances. 
# Convert neighbors to local metres (flat-Earth centered on lead)
coords_m[:, 0] = (neighbor_df["longitude"] - lead_lon) * meters_per_deg_lon
coords_m[:, 1] = (neighbor_df["latitude"] - lead_lat) * meters_per_deg_lat

# Covariance → eigenvalues → anisotropy
cov_matrix = np.cov(centered_coords, rowvar=False)
eig_vals = np.sort(np.linalg.eigvals(cov_matrix))[::-1]
anisotropy = 1.0 - (eig_vals[1] / eig_vals[0])  # 1=line, 0=blob

# Convex hull → density
hull = ConvexHull(coords_m)
density = (len(neighbor_df) / hull.volume) * 1000  # pts/m²×1000

# Spread: RMS distance from centroid
spread = np.sqrt(np.mean(np.sum(centered_coords**2, axis=1)))

# Linearity ratio λ₁/λ₂ — higher = more linear
linearity = eig_vals[0] / (eig_vals[1] + 1e-6)

# Three logistic regime scores
dense_score = sigmoid(0.8*density - 0.03*spread - 1.2*anisotropy)
gully_score = sigmoid(2.0*anisotropy + 0.6*np.log1p(linearity) + 0.4*density)
sparse_score = sigmoid(-1.0*density + 0.04*spread)

1BallTree on installs — Build haversine tree on installed training points only. Per-lead radius query at 100m returns neighbor indices.
2Local projection — Convert neighbor lat/lon to metres via flat-Earth approximation centered on the lead. Accurate to <0.5% within 10 km.
3Covariance eigensplit — 2×2 covariance matrix of projected coords → two eigenvalues. λ₁≈λ₂ = round blob. λ₁≫λ₂ = linear gully. Anisotropy = 1 − λ₂/λ₁.
4Convex hull — scipy.ConvexHull on 2D projected points → area in m². Density = points/area×1000. Spread = RMS distance from centroid.
5Three sigmoid scores — dense_score, gully_score, sparse_score are logistic combinations of the raw descriptors. A lead can trigger multiple regimes (e.g., "dense+gully").
6Regime labeling — Thresholds (0.60, 0.70, 0.40) assign categorical labels. is_sparse and is_deep are binary flags for downstream scoring logic.
What geometry gives that nothing else can: SE tells you "this area converts at 40%." The Gaussian field tells you "the sentiment nearby is positive." Boundary depth tells you "you're safe inside the coverage zone." But none of them tell you about the shape of the evidence. A lead at the end of a narrow service gully (installs only along one road) is qualitatively different from a lead surrounded by installs in all directions (dense neighbourhood), even if SE and field are identical. Anisotropy, linearity, and hull area capture exactly this — the morphological confidence of the local sample.
Why installs only? — When GEOM_INSTALL_FILTER=1, the BallTree is built on installed points only. The shape of the install cloud is what matters for geometry: "how are successful installations distributed around this lead?" Declines are spatially less informative for shape descriptors because they can happen anywhere — a decline 80m away along a road doesn't tell you about the reach of the partner's installation capability in the same way an install does.
